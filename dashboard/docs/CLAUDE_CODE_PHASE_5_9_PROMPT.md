# Caliper Phase 5.9 — Dual-Write Live Events to Aurora

**Paste this entire document into Claude Code running inside the `caliper/` monorepo root.**

---

## 0. Context — the problem

The live `/api/ingest` endpoint writes events to **DynamoDB only**. Aurora's `raw_events` and `raw_assignments` tables are only populated by the synthetic data generator (`dashboard/scripts/generate_demo_data.py`), not by the live API path.

This means when real users click around the headphones site, half the dashboard surfaces don't update:

- DynamoDB-backed surfaces (live stats, CUPED, mSPRT, SRM) — DO update ✅
- Aurora-backed surfaces (dbt segment table, Lift Trend chart, Conversion Rate chart, Funnel, Event Volume chart, Metrics page) — DO NOT update ❌

We're fixing this with a **dual-write** in the ingest API: every event written to DynamoDB also gets written to Aurora `raw_events`. Every assignment gets written to both `caliper-main` and `raw_assignments`.

### Critical constraints

- **The ingest endpoint must stay fast.** DynamoDB writes are hot-path; Aurora writes shouldn't block the response.
- **Failures in Aurora must not break ingest.** If Aurora is slow/down, DynamoDB still gets the events. The user-facing API returns 202 regardless.
- **No duplicate events.** If the API retries, the same event shouldn't appear twice in Aurora. Use `ON CONFLICT DO NOTHING` on a unique key.
- **Preserve all existing functionality.** The DynamoDB-only path must keep working exactly as it does today.

## 2. Approach

### Why dual-write in the API and not a stream-based replication

A more "production-grade" approach would be to subscribe a second Lambda to DynamoDB Streams and have it replicate to Aurora. That's the correct production architecture. But:

- It's overkill for hackathon scope (more infra, more places things break)
- Adds 5-15 second replication latency (visible in the demo)
- The dual-write pattern is fine at our event volume (~10s of events per second peak)

So we dual-write directly from the API. Simple, immediate, no replication lag.

### Make the Aurora write fire-and-forget

The DynamoDB write happens first. The Aurora write fires after, but the API returns 202 without waiting on Aurora. Use Vercel's `waitUntil` to keep the function alive long enough for the Aurora write to complete.

```ts
import { waitUntil } from "@vercel/functions";

// inside the ingest handler:
// 1. Do DynamoDB writes (must succeed, blocks response)
await batchPutEvents(events);

// 2. Fire Aurora write in background
waitUntil(
  insertEventsToAurora(events).catch((err) => {
    console.warn("[ingest] Aurora dual-write failed:", err);
  })
);

return corsResponse({ processed: events.length }, 202);
```

This way:
- DynamoDB write happens immediately, response returns fast
- Aurora write happens in background, doesn't block
- If Aurora fails, log a warning and move on — DynamoDB still has the events
- The user experience is unaffected

### Step 2 — Create Aurora batch insert helpers

In `dashboard/lib/postgres.ts` (or create `dashboard/lib/postgres-batch.ts` if cleaner), add two batch insert functions:

```typescript
import { query } from "./postgres";

interface RawEventInsert {
  experiment_id: string;     // slug or UUID — match how the existing code does it
  user_id: string;
  variant: string;
  event_name: string;
  properties?: Record<string, any>;
  context?: Record<string, any>;
  ts: string | Date;          // ISO 8601
}

interface RawAssignmentInsert {
  experiment_id: string;
  user_id: string;
  variant: string;
  pre_experiment_activity?: number | null;
  assigned_at: string | Date;
}

/**
 * Bulk insert events into raw_events.
 * Uses ON CONFLICT DO NOTHING to be idempotent.
 * For idempotency we need a unique constraint. raw_events has event_id as PRIMARY KEY,
 * which is auto-generated by uuid_generate_v4() — that gives us a natural unique key per insert.
 * So duplicates won't happen unless the same row literally gets inserted twice with the same UUID,
 * which won't occur naturally.
 */
export async function insertRawEvents(events: RawEventInsert[]): Promise<void> {
  if (events.length === 0) return;

  // Build a multi-row INSERT
  // We construct: INSERT INTO raw_events (experiment_id, user_id, ...) VALUES ($1, $2, ...), ($N+1, ...), ...
  // Each row has 7 columns: experiment_id, user_id, variant, event_name, properties, context, ts

  const columns = ["experiment_id", "user_id", "variant", "event_name", "properties", "context", "ts"];
  const placeholders: string[] = [];
  const values: any[] = [];

  events.forEach((e, i) => {
    const offset = i * columns.length;
    const rowPlaceholders = columns.map((_, ci) => `$${offset + ci + 1}`).join(", ");
    placeholders.push(`(${rowPlaceholders})`);
    values.push(
      e.experiment_id,
      e.user_id,
      e.variant,
      e.event_name,
      JSON.stringify(e.properties || {}),
      JSON.stringify(e.context || {}),
      e.ts instanceof Date ? e.ts.toISOString() : e.ts,
    );
  });

  const sql = `
    INSERT INTO raw_events (${columns.join(", ")})
    VALUES ${placeholders.join(", ")}
  `;

  await query(sql, values);
}

/**
 * Insert a single assignment, with ON CONFLICT DO NOTHING.
 * raw_assignments has UNIQUE (experiment_id, user_id), so we get idempotency for free.
 */
export async function insertRawAssignment(assignment: RawAssignmentInsert): Promise<void> {
  await query(
    `INSERT INTO raw_assignments (experiment_id, user_id, variant, pre_experiment_activity, assigned_at)
     VALUES ($1, $2, $3, $4, $5)
     ON CONFLICT (experiment_id, user_id) DO NOTHING`,
    [
      assignment.experiment_id,
      assignment.user_id,
      assignment.variant,
      assignment.pre_experiment_activity ?? null,
      assignment.assigned_at instanceof Date ? assignment.assigned_at.toISOString() : assignment.assigned_at,
    ]
  );
}
```

**Important**: The `experiment_id` column type in `raw_events` and `raw_assignments` is TEXT, and the synthetic generator inserts the experiment **slug** (e.g., `"hero_cta_test"`) — not the UUID. Match that convention. Check `dashboard/scripts/generate_demo_data.py` to confirm — the value passed for `experiment_id` should be `experiment.slug`, not `experiment.id`.

If the existing ingest API works with UUIDs internally, you'll need to look up the slug from the UUID before inserting. The events DynamoDB SK already uses the slug (e.g., `EXP#hero_cta_test`), so this should be straightforward.

### Step 3 — Add Aurora dual-write to the ingest API

In `dashboard/app/api/ingest/route.ts`:

**Add imports:**
```typescript
import { waitUntil } from "@vercel/functions";
import { insertRawEvents } from "@/lib/postgres"; // or wherever you placed the helper
```

**Inside the POST handler**, locate where the events are batch-written to DynamoDB. Right after the `await batchPutEvents(events)` (or similar) call, before the `return` statement, add the Aurora dual-write:

```typescript
// Fire-and-forget Aurora dual-write
// Maps the DynamoDB event shape to the raw_events row shape
const auroraRows = events.map((e: any) => ({
  experiment_id: e.experiment_id,  // should be the slug, not UUID; verify
  user_id: e.user_id,
  variant: e.variant,
  event_name: e.event_name,
  properties: e.properties || {},
  context: e.context || {},
  ts: e.ts || new Date().toISOString(),
}));

waitUntil(
  insertRawEvents(auroraRows).catch((err) => {
    console.warn("[ingest] Aurora dual-write failed:", err);
    // Intentionally not throwing — DynamoDB write already succeeded
  })
);

return corsResponse({ processed: events.length }, 202);
```

**Critical**: Verify the field mapping. Check what shape the events have at this point in the request lifecycle. Common gotchas:

- `experiment_id` might be a UUID. We need the slug for the `raw_events.experiment_id` column. Do a lookup or pass slug down from the assignment.
- `variant` should be a string ("control" or "treatment").
- `ts` might be a number (Unix ms) or string. Convert to ISO 8601.
- `properties` and `context` may be `undefined` — coalesce to `{}`.

## 6. Notes for the implementer

**About the `experiment_id` value**: The existing code in DynamoDB uses the slug as the PK (`EXP#hero_cta_test`), and the synthetic generator inserts the slug into Aurora's `raw_events.experiment_id`. Match that. If the ingest endpoint currently has the UUID and needs the slug, look it up from the experiments table once at the start of the handler and pass the slug into the Aurora write.

**About the connection pool**: PostgreSQL has connection limits. If the dual-write opens a new connection per request, we could exhaust the pool under load. Verify `lib/postgres.ts` uses a connection pool (likely with `pg.Pool`). If not, ensure connections are released after each query.

**About the dbt schedule**: The dbt Lambda runs every 15 minutes via EventBridge. After this dual-write is live, every 15 minutes the segment table will refresh with the live event data. So `mart_segment_results` will lag the actual events by up to 15 minutes — that's expected and fine for the demo.

**About performance**: At our event volume (peak ~10 events/sec from a few demo users) the Aurora dual-write is well within Postgres capacity. If we ever needed to scale, we'd switch to streaming via DynamoDB → Lambda → Aurora, but that's a post-hackathon concern.
